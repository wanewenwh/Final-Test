# 03 详细设计

## 3.1 FileStorage 类

### 3.1.1 属性

| 属性 | 类型 | 说明 |
|------|------|------|
| `report_dir` | str | 报告存储目录路径 |

### 3.1.2 方法

#### `__init__(report_dir="reports")`
初始化存储目录。若目录不存在则自动创建。

#### `save_report(question, ai_response) -> filename`
保存报告文件。

**参数：**
- `question`: 用户提问文本
- `ai_response`: AI 返回的报告内容（或错误信息）

**文件命名规则：** `科学遐思_YYYYMMDD_HHMMSS.txt`

**文件内容格式：**
```
【科学遐思提问】
{用户问题}
========================================
【AI 研究报告】
{AI 返回内容}
```

**异常处理：** I/O 异常由外层捕获，不在此方法内处理。

#### `list_reports() -> list[dict]`
返回报告列表，按时间倒序。

**返回格式：**
```python
[
    {"name": "科学遐思_20240101_120000.txt", "time": "2024-01-01 12:00:00", "size": 2048},
    ...
]
```

**安全措施：** 使用 `os.listdir` 避免递归，仅扫描指定目录。

#### `get_report(filename) -> str`
读取指定报告文件内容。

**安全措施：** 使用 `os.path.basename` 防止路径穿越攻击。

## 3.2 AIService 类

### 3.2.1 属性

| 属性 | 类型 | 说明 |
|------|------|------|
| `api_key` | str | DeepSeek API Key |
| `model` | str | 模型名称，默认 deepseek-chat |
| `base_url` | str | API 端点 URL |
| `SYSTEM_PROMPT` | str (类常量) | 固定的系统提示词 |

### 3.2.2 方法

#### `__init__(api_key=None, model="deepseek-chat")`
初始化 AI 服务。API Key 可从参数传入或从环境变量 `DEEPSEEK_API_KEY` 读取。

#### `generate_report_async(user_text, on_complete, on_error) -> task_id`
异步生成报告。

**流程：**
1. 生成唯一 `task_id`（基于时间戳+线程ID）
2. 创建后台 `threading.Thread` 执行 `_do_generate`
3. 线程设为 `daemon=True`，防止阻塞主进程退出
4. 立即返回 `task_id`

#### `_do_generate(task_id, user_text, on_complete, on_error)`
后台线程执行体。

**流程：**
1. 调用 `_call_api(user_text)`
2. 成功 → 触发 `on_complete` 回调
3. 失败 → 触发 `on_error` 回调，传入错误信息

#### `_call_api(user_text) -> str`
实际调用 DeepSeek API。

**请求参数：**
```
model: deepseek-chat
temperature: 0.7
max_tokens: 2048
messages:
  - role: system → SYSTEM_PROMPT
  - role: user   → "科学遐思：{user_text}"
```

**降级策略：** 若 API Key 为空或为默认值 `your-api-key-here`，使用 `_mock_response()` 返回模拟数据。

#### `_mock_response(user_text) -> str`
生成模拟 AI 回复（用于演示和无 API Key 环境）。

## 3.3 app.py 路由设计

### 路由表

| 方法 | 路径 | 参数 | 返回 |
|------|------|------|------|
| GET | `/` | — | HTML 页面 |
| POST | `/api/send_to_ai` | `{"text": "..."}` | `{"task_id": "...", "status": "generating"}` |
| GET | `/api/reports` | — | `{"reports": [...]}` |
| GET | `/api/reports/<filename>` | — | `{"content": "..."}` |
| GET | `/api/status/<task_id>` | — | `{"status": "completed"/"generating"/"failed"}` |

### 任务状态管理

使用线程安全的字典 `_task_status` 存储任务状态，读写操作受 `threading.Lock` 保护。

**状态转换：**
```
generating → completed (成功)
generating → failed (失败)
```

## 3.4 前端交互流程

```
用户操作                   前端逻辑                    后端逻辑
─────────               ──────────                ──────────
1. 输入灵感碎片
2. 点击暂存
    ─────────→  stashTexts.push(text)               
                刷新暂存区显示
3. 重复1-2
4. 点击发送
    ─────────→  POST /api/send_to_ai  ────────→  启动后台线程
                清空暂存区                         返回task_id
                显示"生成中..."
    ←─────────  {"task_id": "..."}
5. 继续输入新想法（不阻塞）
6. 轮询状态
    ─────────→  GET /api/status/task_id  ───────→  查询状态
    ←─────────  {"status": "completed"}
                刷新报告列表
7. 点击报告
    ─────────→  GET /api/reports/xxx.txt ──────→  读取文件
                弹出模态框显示内容
```

## 3.5 数据结构选择理由

| 数据 | 结构 | 理由 |
|------|------|------|
| 暂存区 | `list[str]` | 需要顺序追加、整体读取，列表的 append/join 操作均为 O(1)/O(n) |
| 任务状态 | `dict[str, dict]` | 以 task_id 为键快速查找，字典查询 O(1) |
| 报告列表 | `list[dict]` | 需要排序、筛选，列表操作灵活 |
| 报告内容 | str (TXT) | 纯文本存储，无需数据库，便于直接查看 |
